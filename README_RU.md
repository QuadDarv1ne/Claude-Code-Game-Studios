## Зачем это нужно

**Создавать игру в одиночку с помощью ИИ** — это мощно, но у одного чата нет структуры.

Никто не остановит вас от хардкода магических чисел, пропуска дизайн-документов или написания спагетти-кода. Нет этапа QA, ревью дизайна, никто не спросит «действительно ли это соответствует видению игры?»

**Claude Code Game Studios** решает эту проблему, придавая вашей ИИ-сессии структуру реальной студии. Вместо одного универсального помощника вы получаете 48 специализированных агентов, организованных в иерархию студии — директоров, которые охраняют видение, руководителей отделов, которые отвечают за свои области, и специалистов, которые выполняют практическую работу.

У каждого агента есть определённые обязанности, пути эскалации и контрольные точки качества.

**Результат:** вы по-прежнему принимаете все решения, но теперь у вас есть команда, которая задаёт правильные вопросы, ловит ошибки на раннем этапе и поддерживает порядок в проекте от первого брифа до релиза.

## Начало работы

### Требования

- [Git](https://git-scm.com/)
- [Claude Code](https://docs.anthropic.com/en/docs/claude-code) (`npm install -g @anthropic-ai/claude-code`)
- **Рекомендуется**: [jq](https://jqlang.github.io/jq/) (для валидации хуков) и Python 3 (для валидации JSON)

Все хуки работают в штатном режиме, если отсутствуют опциональные инструменты — ничего не ломается, вы просто теряете валидацию.

### Настройка

1. **Склонируйте или используйте как шаблон**:
   ```bash
   git clone https://github.com/Donchitos/Claude-Code-Game-Studios.git my-game
   cd my-game
   ```

2. **Откройте Claude Code** и начните сессию:
   ```bash
   claude
   ```

3. **Запустите `/start`** — система спросит, где вы находитесь (нет идей, смутная концепция, чёткий дизайн, существующая работа) и направит к правильному рабочему процессу. Никаких предположений.

   Или перейдите сразу к конкретному навыку, если уже знаете, что вам нужно:
   - `/brainstorm` — исследование идей игры с нуля
   - `/setup-engine godot 4.6` — настройка движка, если вы уже определились
   - `/project-stage-detect` — анализ существующего проекта

## Как это работает

### Координация агентов

Агенты следуют структурированной модели делегирования:

1. **Вертикальное делегирование** — директоры делегируют руководителям, руководители делегируют специалистам
2. **Горизонтальная консультация** — агенты одного уровня могут консультировать друг друга, но не могут принимать обязательные межотдельские решения
3. **Разрешение конфликтов** — разногласия эскалируются к общему родителю (`creative-director` для дизайна, `technical-director` для технических вопросов)
4. **Распространение изменений** — межотдельские изменения координируются `producer`
5. **Границы областей** — агенты не модифицируют файлы вне своей области без явного делегирования

### Совместная работа, а не автономность

Это **не** автопилот. Каждый агент следует строгому протоколу сотрудничества:

1. **Спросить** — агенты задают вопросы перед предложением решений
2. **Представить варианты** — агенты показывают 2-4 варианта с плюсами и минусами
3. **Вы решаете** — пользователь всегда принимает решение
4. **Черновик** — агенты показывают работу перед финализацией
5. **Подтверждение** — ничего не записывается без вашего одобрения

Вы остаётесь у руля. Агенты обеспечивают структуру и экспертизу, а не автономию.

### Автоматическая безопасность

**Хуки** запускаются автоматически в каждой сессии:

| Хук | Триггер | Что делает |
|-----|---------|------------|
| `validate-commit.sh` | `git commit` | Проверка на хардкод значений, формат TODO, валидность JSON, секции дизайн-документов |
| `validate-push.sh` | `git push` | Предупреждение о пушах в защищённые ветки |
| `validate-assets.sh` | Запись файлов в `assets/` | Проверка соглашений об именовании и структуры JSON |
| `session-start.sh` | Открытие сессии | Загрузка контекста спринта и недавней активности git |
| `detect-gaps.sh` | Открытие сессии | Обнаружение свежих проектов (предлагает `/start`) и отсутствующей документации при наличии кода/прототипов |
| `pre-compact.sh` | Сжатие контекста | Сохранение заметок о прогрессе сессии |
| `session-stop.sh` | Закрытие сессии | Журнал достижений |
| `log-agent.sh` | Вызов агента | Аудит всех вызовов субагентов |

**Правила разрешений** в `settings.json` автоматически разрешают безопасные операции (git status, запуск тестов) и блокируют опасные (force push, `rm -rf`, чтение `.env` файлов).

### Правила, привязанные к путям

Стандарты кодирования автоматически применяются в зависимости от расположения файлов:

| Путь | Применяет |
|------|-----------|
| `src/gameplay/**` | Значения, управляемые данными, использование delta time, нет ссылок на UI |
| `src/core/**` | Нулевые аллокации в горячих путях, потокобезопасность, стабильность API |
| `src/ai/**` | Бюджеты производительности, отладка, управляемые данными параметры |
| `src/networking/**` | Сервер-авторитетность, версионирование сообщений, безопасность |
| `src/ui/**` | Нет владения состоянием игры, готовность к локализации, доступность |
| `design/gdd/**` | Обязательные 8 секций, формат формул, граничные случаи |
| `tests/**` | Именование тестов, требования к покрытию, паттерны фикстур |
| `prototypes/**` | Ослабленные стандарты, требуется README, задокументированная гипотеза |

## Философия дизайна

Этот шаблон основан на профессиональных практиках разработки игр:

- **MDA Framework** — Механики, Динамика, Эстетика для дизайна игры
- **Теория самодетерминации** — Автономия, Компетентность, Связанность для мотивации игрока
- **Дизайн состояния потока** — Баланс сложности и навыка для вовлечённости игрока
- **Типы игроков Бартла** — Таргетирование и валидация аудитории
- **Разработка, управляемая верификацией** — Сначала тесты, потом реализация

## Настройка

Это **шаблон**, а не жёсткий фреймворк. Всё предназначено для настройки:

- **Добавить/удалить агентов** — удалите ненужные файлы агентов, добавьте новые для своих областей
- **Редактировать промпты агентов** — настройте поведение агентов, добавьте специфику проекта
- **Изменить навыки** — адаптируйте рабочие процессы под процесс вашей команды
- **Добавить правила** — создайте новые правила, привязанные к путям, для структуры каталогов вашего проекта
- **Настроить хуки** — измените строгость валидации, добавьте новые проверки
- **Выбрать движок** — используйте набор агентов Godot, Unity или Unreal (или никакой)

## Поддержка платформ

Протестировано на **Windows 10** с Git Bash. Все хуки используют POSIX-совместимые паттерны (`grep -E`, а не `grep -P`) и включают резервные варианты для отсутствующих инструментов. Работает на macOS и Linux без изменений.
